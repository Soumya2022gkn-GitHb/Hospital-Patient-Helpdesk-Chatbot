# Phase 10: RAG Chain

## Hospital Patient Helpdesk Chatbot

**Python module:** `06_rag_pipeline/10_rag_chain.py`  
**Jupyter notebook:** `13_notebooks/10_rag_chain.ipynb`  
**Purpose:** Combine retrieval, safe prompt construction, and provider-neutral generation to produce final chatbot answers with citations and source provenance.

---

## 1. Phase objective

Phase 10 is the orchestration layer of the RAG workflow. It does not reimplement retrieval, prompts, or model access. Instead, it coordinates the public interfaces from:

1. Phase 7: retrieve and rank approved hospital evidence.
2. Phase 8: select the correct safety mode and construct a citation-ready prompt.
3. Phase 9: generate and validate a provider-neutral answer.

The final result is a stable answer contract for the future FastAPI backend and Streamlit interface.

## 3. Input files

### `01_data/sample_queries/test_questions.csv`

Contains synthetic helpdesk questions used for end-to-end validation. The `question` column is required. Existing category, expected-source, and safety-class columns remain useful for evaluation context.

### `05_vector_store/chroma_db/06_vector_index.sqlite3`

The Phase 6 vector index stores approved chunks, float32 embeddings, and retrieval metadata. Phase 7 validates its model, dimension, backend, schema version, and database integrity before querying it.

### `02_config/prompt_config.yaml`

Defines the base hospital assistant instruction and the approved insufficient-context message. Phase 8 adds source delimiters, prompt-injection defenses, citation rules, emergency routing, and unsafe-medical-advice refusal instructions.

### `.env`

Controls the Phase 9 provider and model. The default is `offline`, which makes the included validation deterministic and private. Hosted credentials are never placed in Phase 10 artifacts.

## 5. Final answer schema

Each item in `10_rag_answers.json` follows this contract:

```json
{
  "answer_id": "ANSWER-001",
  "chain_version": "1.0",
  "question": "How can I book an appointment?",
  "answer": "Use the patient portal ... [S1]",
  "mode": "grounded_answer",
  "citations": ["[S1]"],
  "sources": [
    {
      "citation": "[S1]",
      "chunk_id": "...",
      "source_file": "faqs/hospital_faqs.json",
      "source_type": "json",
      "department": "Portal Support",
      "content_category": "appointments",
      "page_reference": null,
      "score": 0.74580427
    }
  ],
  "retrieval_confidence": "high",
  "safety_labels": [],
  "provider": "offline",
  "model": "offline-grounded-v1",
  "finish_reason": "stop",
  "timings": {
    "retrieval_ms": 4.0,
    "prompt_ms": 0.5,
    "generation_ms": 0.5,
    "total_ms": 5.0
  }
}
```

`citations` contains labels actually used in the answer. `sources` contains the approved evidence supplied to the model, including sources that were available but not cited in the final text.

## 6. End-to-end execution flow

```text
Patient question
      |
      v
Phase 7 retriever
  - embed question
  - search vector index
  - apply filters and reranking
  - assign confidence and safety labels
      |
      v
Phase 8 prompt builder
  - choose safety mode
  - budget and label sources
  - add grounding and injection defenses
      |
      v
Phase 9 LLM client
  - call configured provider
  - normalize provider response
  - validate citations and safety behavior
      |
      v
Phase 10 final answer
  - combine answer and provenance
  - validate the complete contract
  - record stage timings
```

## 7. Python module code sections

### Chain configuration

`RAGChainConfig` validates the retrieval count, candidate pool, hybrid vector weight, prompt source budget, context character budget, and minimum confidence. `build_stage_configs()` translates those values into the native Phase 7 and Phase 8 configuration objects.

### Final response contracts

`AnswerSource`, `StageTimings`, and `RAGAnswer` define the data returned to later API and UI phases. `ChainRunResult` identifies all generated artifacts and run totals.

### Dependency loading

`import_module()` safely imports numbered Python files by path and registers them in `sys.modules`, which supports their dataclasses. `load_dependencies()` loads Phases 7, 8, and 9 from the project folder.

### Source mapping

`source_records()` converts Phase 8 prompt sources into the public Phase 10 provenance schema. It preserves the citation label, chunk ID, source file, source type, department, category, optional page reference, and ranking score.

### One-question orchestration

`answer_question()` is the central chain function. It:

1. normalizes and validates the question;
2. retrieves evidence through Phase 7;
3. constructs and validates the prompt through Phase 8;
4. creates the configured provider through Phase 9;
5. generates and validates the LLM response;
6. combines answer text and source provenance; and
7. validates the complete final contract.

Retrieval, prompt, generation, and total time are measured separately.

### Final validation

`validate_chain_answer()` rejects empty questions or answers, unknown citations, uncited grounded answers, missing emergency routing, missing unsafe-advice refusal, and negative timing values.

### Batch evaluation

`run_chain_evaluation()` loads shared dependencies once, processes every sample question independently, writes successful answers and audit rows, captures sanitized failures, creates plots, and writes the aggregate report.

Per-question isolation prevents one bad record or temporary provider failure from discarding successful answers.

### Diagnostics

`generate_plots()` creates:

- a stacked latency plot that separates retrieval, prompt, and generation cost; and
- an answer-mode plot that confirms grounding and safety routes.

### Command-line interface

Run the full offline sample evaluation:

```powershell
python 06_rag_pipeline/10_rag_chain.py
```

Run one question:

```powershell
python 06_rag_pipeline/10_rag_chain.py --question "Where is cardiology?"
```

Use a configured provider:

```powershell
python 06_rag_pipeline/10_rag_chain.py --provider openai
```

Review privacy authorization and provider cost before using hosted models.

## 9. Notebook versus Python module

| Topic | `10_rag_chain.ipynb` | `10_rag_chain.py` |
|---|---|---|
| Primary role | Interactive learning, trace inspection, safety demonstration, and visual validation. | Reusable orchestration for API, UI, tests, and automation. |
| Business logic | Imports and calls the module. | Owns chain configuration, orchestration, validation, batching, artifacts, and CLI. |
| Typical execution | Run cells in order. | Import `answer_question()` or run from the command line. |
| Provider in examples | Explicitly forced to `offline`. | Uses `.env` or `--provider`. |
| Outputs | Displays representative records, reports, and plots inline. | Writes numbered JSON, CSV, and PNG artifacts. |
| Failure behavior | Assertions stop the visible walkthrough. | Isolates and records failures per question during batch execution. |

The notebook and module stay compatible because the notebook does not duplicate chain logic.

## 11. Healthcare safety and privacy

- The chain must not diagnose disease, prescribe treatment, or recommend medication dosage.
- Possible emergencies must be directed to local emergency services immediately.
- Answers must remain grounded in approved hospital evidence and preserve traceable citations.
- Retrieval confidence is an information-retrieval signal, not medical certainty.
- Protected health information must not be sent to external providers without appropriate authorization, contracts, controls, and necessity.
- Logs and failure records should minimize patient text and never contain API keys.
- Phase 11 guardrails, evaluation, monitoring, and human governance remain necessary before deployment.

Phase 10 provides an end-to-end hospital information workflow. It is not clinical decision support and does not replace qualified healthcare professionals.
