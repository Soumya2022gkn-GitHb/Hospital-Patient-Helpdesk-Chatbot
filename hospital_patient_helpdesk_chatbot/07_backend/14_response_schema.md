# Phase 14: Response Schema

## Hospital Patient Helpdesk Chatbot

**Python module:** `07_backend/14_response_schema.py`  
**Jupyter notebook:** `13_notebooks/14_response_schema.ipynb`  
**Purpose:** Define and validate the chatbot response format with answer, sources, confidence, and safety flag.

---

## 1. Phase objective

Phase 14 formalizes the response contract returned by the backend `/chat` endpoint. It validates real Phase 12 API responses and deliberately malformed adversarial examples so downstream UI, logging, evaluation, and monitoring code can rely on a stable schema.

The module is framework-independent and can run without Pydantic or FastAPI. When Pydantic is installed, it can create response models suitable for FastAPI/OpenAPI.

## 3. Response fields

| Field | Rule | Purpose |
|---|---|---|
| `request_id` | Expected format like `REQ-ABCDEF123456` | Correlates one API response. |
| `answer` | Required, non-empty, max 2500 characters | Patient-facing answer text. |
| `mode` | Known response mode | Indicates grounded, emergency, unsafe-advice, insufficient-context, or security-refusal behavior. |
| `citations` | List of labels such as `[S1]` | Labels actually used in the answer. |
| `sources` | Up to 6 source records | Approved provenance supplied with the response. |
| `retrieval_confidence` | `none`, `low`, `medium`, or `high` | Retrieval quality signal, not medical certainty. |
| `safety_flag` | Boolean | True when guardrails changed or blocked output. |
| `guardrail_action` | `pass`, `override`, `block`, or `redact` | Final Phase 11 action. |
| `risk_level` | `low`, `medium`, `high`, or `critical` | Guardrail policy severity. |
| `triggered_rules` | List of rule IDs | Explains safety action when flagged. |
| `provider` and `model` | Required text | Traceability for generation backend. |
| `latency_ms` | Non-negative number | End-to-end API service latency. |
| `timestamp_utc` | ISO-like timestamp | Response creation time. |

## 4. Source fields

Each source includes:

- `citation`
- `source_file`
- optional `chunk_id`
- optional `source_type`
- optional `department`
- optional `content_category`
- optional `page_reference`
- optional numeric `score`

Every citation in `citations` must appear in the response's own `sources` list.

## 7. Python module code sections

### Constants and contracts

The module defines accepted modes, confidence labels, guardrail actions, risk levels, answer length limit, source limit, request-ID pattern, citation pattern, and timestamp pattern.

`ResponseValidationConfig`, `ResponseSource`, `ChatResponse`, `ResponseValidationResult`, and `ResponseSchemaRunResult` provide typed, reusable boundaries.

### Source validation

`validate_source()` checks citation format, required `source_file`, numeric non-negative score, and optional metadata normalization.

### Response validation

`validate_response_payload()` verifies required fields, answer length, recognized mode, confidence, guardrail action, risk level, provider, model, latency, timestamp shape, citation list, source list, triggered-rule list, citation-to-source integrity, grounded citation requirements, and safety-flag consistency.

### Optional Pydantic model

`create_pydantic_response_model()` lazily imports Pydantic and returns a nested response model for FastAPI/OpenAPI. If Pydantic is unavailable, the function raises a clear installation message while core validation still works.

### Batch validation

`run_response_schema_validation()` loads real Phase 12 responses, appends adversarial malformed payloads, validates everything, writes JSON and CSV artifacts, creates plots, and returns output paths and counts.

## 9. Notebook versus Python module

| Topic | `14_response_schema.ipynb` | `14_response_schema.py` |
|---|---|---|
| Main purpose | Interactive response-contract review. | Reusable canonical response validation. |
| Validation logic | Imports module functions. | Owns dataclasses, source checks, citation checks, safety consistency, artifacts, plots, and CLI. |
| Inputs | Uses Phase 12 API responses and inline adversarial examples. | Accepts any compatible API response JSON file. |
| Outputs | Displays examples, report, and plots. | Writes `14_` JSON, CSV, and PNG artifacts. |
| Dependency behavior | Reports whether Pydantic is installed. | Works without Pydantic; creates model only when installed. |

The notebook does not duplicate validation rules, so both files remain aligned.

## 12. Safety and privacy notes

- Response-schema validation does not replace Phase 11 safety guardrails.
- Retrieval confidence is not medical certainty.
- Safety/refusal responses usually should not cite retrieved sources because their text is policy-driven.
- Audit output avoids storing full answer text.
- Response validation helps downstream UI, evaluation, and monitoring code fail early when the contract changes.
